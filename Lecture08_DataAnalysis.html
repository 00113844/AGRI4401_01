Today's session focuses on practical tools that leverage Large Language Models (LLMs) to accelerate your data analysis work in R. We'll cover two primary approaches: APPROACH 1 — GitHub Copilot: An AI pair programmer that provides real-time code suggestions as you type in RStudio, helping write functions, generate boilerplate code, and answer questions through comments. Both tools complement each other: Copilot excels at inline autocomplete and quick suggestions, while Ellmer handles more complex conversational queries and explanations. We'll walk through complete setup procedures including API key management, demonstrate practical usage patterns, and establish best practices for incorporating LLM assistance into your precision agriculture data analysis workflow without compromising code quality or understanding.

Copilot's "ghost text" interface feels magical when it works well but requires understanding how to guide it effectively. GHOST TEXT MECHANICS: As you type, Copilot analyzes context and generates suggestions displayed in light grey text extending from your cursor. This happens INSTANTLY as you type — no explicit invocation required. If suggestion looks correct, pressing Tab accepts it entirely. If partially correct, accept then edit. If wrong, ignore and keep typing. CYCLING ALTERNATIVES: Copilot typically generates multiple suggestion candidates. Use Alt+] (next suggestion) or Alt+[ (previous suggestion) to cycle through alternatives without accepting. This is useful when first suggestion is close but not quite right — often the second or third option is better. COMMENT-DRIVEN DEVELOPMENT: Copilot's most powerful feature is translating comments into code. Example workflow for yield analysis: 1. Write comment: "# Load yield data and filter outliers beyond 3 standard deviations" 2. Press Enter → Copilot suggests complete code block loading CSV, calculating SD, applying filter 3. Review suggestion (verify column names match your data) 4. Accept with Tab or modify as needed This works because Copilot recognizes common agricultural data patterns from its training on public code, including precision agriculture repositories. COPILOT AS LEARNING TOOL: When Copilot suggests unfamiliar syntax (e.g., you've never used tidyr::pivot_wider), this is learning opportunity. Accept the suggestion, run it, observe output, then look up documentation to understand WHY it works. Over time, you internalize these patterns and become less dependent on suggestions. WHEN COPILOT STRUGGLES: Suggestions are weakest with: (1) novel or domain-specific algorithms not widely published, (2) code requiring deep understanding of data structure Copilot can't infer from context, (3) complex multi-step analyses where intermediate variables aren't clearly named. In these cases, break problem into smaller steps with explicit comments guiding each step.

Asking Questions with GitHub Copilot

  • Question syntax: # q: [your question]?
  • Answer format: Copilot responds with # a: [answer]
  • Use cases: Syntax help, debugging, best practices
  • Example questions:
    • "How to join two data frames by multiple columns in dplyr?"
    • "What's causing this 'subscript out of bounds' error?"
    • "Best way to handle missing values in spatial data?"
# q: How to calculate spatial autocorrelation in R?
# a: Use the 'spdep' package: 
# library(spdep)
# coords <- cbind(data$x, data$y)
# nb <- knn2nb(knearneigh(coords, k=4))
# moran.test(data$yield, nb2listw(nb))
The question-answer syntax transforms Copilot from autocomplete tool into interactive Q&A assistant. SYNTAX DETAILS: The "# q:" prefix signals Copilot that this is a question requiring explanation, not code to complete. Question mark at end is optional but improves response quality. Copilot responds with "# a:" comment containing answer, often followed by example code demonstrating the concept. QUALITY OF ANSWERS: Copilot's answers are better for: (1) syntax and API questions ("How do I use function X?"), (2) common programming patterns ("Best way to accomplish task Y?"), (3) error explanations when error message is included in question. Answers are weaker for: (1) conceptual statistical questions requiring judgment ("Should I use linear or nonlinear model?"), (2) domain-specific agricultural decisions ("Is this yield level normal for sandy soil?"), (3) questions requiring context Copilot can't infer from current file. PRACTICAL WORKFLOW INTEGRATION: When stuck during analysis, try asking Copilot before searching online. Example scenario: You're creating spatial variability map but forgot ggplot syntax for equal coordinate aspect ratio. Instead of googling, type: # q: How to make ggplot use equal x/y axis scales? # a: Add coord_equal() or coord_fixed() layer This is faster than web search and keeps you in flow state within RStudio. COMBINING WITH WEB SEARCH: For complex questions, use hybrid approach: (1) Ask Copilot for quick answer, (2) if insufficient, use answer to formulate better web search query with correct terminology. Copilot's terminology often reveals what to search for. CAVEAT ON ACCURACY: Copilot occasionally generates plausible-sounding but incorrect answers (hallucination problem common to LLMs). Always verify suggestions against official documentation or test with small example before applying to full dataset. Treat answers as "probably right but confirm" not "definitely correct."

Creating Google Cloud Project & API Key

  1. Create project: Visit Google Cloud Console
  2. Enable Generative AI API: Navigate to APIs & Services
  3. Generate API key: Go to AI Studio API Keys
  4. Copy key: Securely store the generated key (appears only once)
  5. Never share: Treat as password — do not commit to Git or share publicly

Your API key grants access to cloud resources charged to your account

API key generation requires several steps through Google's developer interfaces. Work through these carefully. STEP 1 — PROJECT CREATION: Google Cloud organizes resources into "projects" — think of these as containers for related APIs and billing. For AGRI4401, create project named "AGRI4401-Analysis" or similar. You'll need Google account (your UWA email works if linked to Google). STEP 2 — ENABLE API: After creating project, you must explicitly enable "Generative Language API" (Gemini's underlying service). In Cloud Console, search for "Generative Language API" in API Library, click Enable button. This authorizes your project to call Gemini endpoints. STEP 3 — GENERATE KEY: Visit AI Studio (aistudio.google.com/app/apikey) while logged into the same Google account. Click "Get API key" → "Create API key in new project" (or select existing project if you created one in step 1). System generates key immediately. STEP 4 — COPY AND STORE: KEY APPEARS ONLY ONCE. After clicking "Copy," key is yours to use. If you lose it, you cannot retrieve — must generate new key and revoke old one. Store immediately in password manager or secure note. DO NOT paste into Slack, email, or any shared document. SECURITY IMPLICATIONS: Anyone with your API key can make requests to Google's services under your account, potentially incurring charges if you exceed free tier. If key is ever compromised (accidentally committed to public GitHub repo, shared in screenshot), immediately revoke it in API console and generate replacement. BEST PRACTICE: Generate separate keys for different purposes (one for coursework, separate one for research projects). This limits damage if one key is compromised and makes key rotation easier. NEXT STEP: After copying key, we'll store it securely in R environment variables (next slide) so R scripts can access without hardcoding key in code.

Securely Storing API Keys in R

# Install usethis package if needed
install.packages("usethis")

# Open .Renviron file for editing
usethis::edit_r_environ()

Add your key to the opened .Renviron file:

GEMINI_API_KEY="AIza...your_actual_key_here"

Save file, restart RStudio, then access key in code:

# Key is now available via Sys.getenv()
api_key <- Sys.getenv("GEMINI_API_KEY")
Environment variables provide secure, version-control-safe method for storing sensitive credentials. WHY .Renviron FILE: R's .Renviron file is a hidden text file in your home directory that R reads at startup, loading key-value pairs as environment variables. This keeps secrets OUT of your code files (which might be committed to Git) while making them available to all R sessions. USETHIS PACKAGE: The usethis::edit_r_environ() function is convenience wrapper that: (1) locates correct .Renviron file (typically ~/.Renviron on Mac/Linux or Documents\.Renviron on Windows), (2) creates file if it doesn't exist, (3) opens in RStudio editor for easy editing. Without usethis, you'd need to manually locate and create this hidden file. SYNTAX DETAILS: In .Renviron file, format is VARIABLE_NAME="value" with NO spaces around = sign. Variable names conventionally UPPERCASE_WITH_UNDERSCORES. Quotes around value are optional for simple strings but recommended for API keys which may contain special characters. MULTIPLE KEYS: You can store multiple keys in same .Renviron file: GEMINI_API_KEY="AIza..." OPENAI_API_KEY="sk-..." DATABASE_PASSWORD="secret123" Each on separate line. This becomes your personal credential vault for all R projects. RESTART REQUIRED: After editing .Renviron, must restart R session (Session → Restart R in RStudio) to load new variables. This is ONE-TIME setup — afterwards, variables available automatically in every new R session. ACCESSING IN CODE: Sys.getenv("VARIABLE_NAME") retrieves stored value. Returns empty string "" if variable not found (useful for error checking). Example defensive code: api_key <- Sys.getenv("GEMINI_API_KEY") if (api_key == "") { stop("GEMINI_API_KEY not found in environment. Did you set it in .Renviron?") } SECURITY BENEFIT: Now you can share your R scripts publicly (GitHub, with classmates) without exposing credentials. Anyone running your code needs their own .Renviron setup with their own keys. GIT CONFIGURATION: .Renviron file is automatically ignored by Git (part of default .gitignore patterns) so won't be accidentally committed. Double-check with `git status` before committing if you're paranoid.

Introducing Ellmer: LLM Assistant in RStudio

  • What it is: R package providing interactive chat interface to Google Gemini
  • Use cases: Ask questions, request code snippets, get explanations
  • Complements Copilot: Chat-based vs inline suggestions
  • Installation: Available on CRAN
install.packages("ellmer", dependencies = TRUE)
Ellmer (pronounced "elmer") bridges gap between Copilot's inline suggestions and full conversational LLM interface. PACKAGE BACKGROUND: Developed by Hadley Wickham and Posit team (creators of RStudio, tidyverse packages) as official R interface to major LLM providers. Currently supports Google Gemini (via API we just set up) with plans to support Anthropic Claude and other models. COPILOT VS ELLMER COMPARISON: COPILOT STRENGTHS: - Real-time code completion as you type - Zero-latency suggestions (feels instant) - Great for writing boilerplate and repetitive code - No explicit API call (happens automatically) ELLMER STRENGTHS: - Conversational context (multi-turn dialogues) - Longer, more detailed explanations - Can attach code snippets or data samples to questions - Better for exploratory "what should I do?" questions PRACTICAL USAGE PATTERN: Use Copilot as primary coding assistant (90% of time), but when you hit conceptual roadblock or need to plan complex analysis, open Ellmer chat for deeper consultation, then return to Copilot for implementation. INSTALLATION NOTE: Specify dependencies = TRUE to ensure all required packages installed (includes httr2 for HTTP requests, jsonlite for API response parsing, rstudioapi for browser integration). FIRST-TIME SETUP: After installation, Ellmer automatically detects GEMINI_API_KEY from environment (set in previous slide). If you properly configured .Renviron, no additional authentication needed — Ellmer just works. VERSION COMPATIBILITY: Requires RStudio version 2023.06.0 or newer for integrated browser window. On older RStudio versions, Ellmer falls back to external browser window (functional but less seamless experience).

Starting an Ellmer Chat Session

library(ellmer)

# Create Gemini chat session
chat <- chat_google_gemini()

# Launch browser-based chat interface
live_browser(chat)
  • Browser window: Opens in RStudio Viewer pane or system browser
  • Persistent context: Chat remembers previous exchanges within session
  • Code snippets: Can paste data/code for analysis
  • Export: Copy responses back to R script
These three lines of code launch fully interactive LLM chat interface within RStudio environment. STEP-BY-STEP EXECUTION: LINE 1 — library(ellmer): Loads package functions into current session. Must run this in every new R session (or add to startup script). LINE 2 — chat <- chat_google_gemini(): Creates chat object representing conversation with Gemini model. This object stores conversation history (previous messages and responses) enabling contextual follow-up questions. The chat object uses your GEMINI_API_KEY from environment automatically. LINE 3 — live_browser(chat): Opens interactive chat interface. In RStudio 2023.06+, appears in Viewer pane (lower-right by default). In older versions or if viewer unavailable, opens system default web browser with localhost URL. CHAT INTERFACE FEATURES: - Text input box at bottom for typing questions - Message history scrolls up as conversation progresses - Code blocks in responses have "Copy" button for easy transfer to R script - Session persists until you close RStudio or explicitly end chat EXAMPLE CONVERSATION FLOW: You: "I have yield data with outliers. What's the best approach to identify and handle them?" Gemini: "Several approaches depending on your goals: (1) Visual inspection with boxplots, (2) Statistical thresholds like 1.5×IQR or ±3SD, (3) Spatial filtering for geographically isolated anomalies. For precision ag data, I recommend combining approaches..." [Includes code examples for each method] You: "Show me the spatial filtering approach in more detail." Gemini: [Generates complete code example using spdep package for spatial outlier detection] Notice second question builds on first — Gemini remembers context that we're discussing yield outliers. PRACTICAL TIPS: - Include example data in questions: "Here's structure of my data: str(yield_df)" followed by your question improves response relevance - Ask for clarification: "I don't understand step 3, can you explain differently?" works better than giving up - Request code style: "Write this using tidyverse syntax" vs "Use base R" tailors responses to your preference - Export useful responses: Copy code blocks to .R file with comment acknowledging LLM origin for future reference CONVERSATION MANAGEMENT: Chat history persists in `chat` object. To start fresh topic, create new chat object: `chat2 <- chat_google_gemini()`. Original chat remains available if you want to return to it.

Practical Ellmer Use Cases for Data Analysis

  • Statistical guidance: "Which test for comparing means across 5 treatment zones?"
  • Code explanation: Paste complex code, ask "What does this do?"
  • Debugging help: Share error message, request troubleshooting steps
  • Data exploration ideas: "What visualizations would reveal spatial patterns?"
  • Best practices: "How should I handle missing GPS coordinates in yield data?"
Ellmer excels at different types of questions than simple code completion. These use cases showcase its strengths. USE CASE 1 — STATISTICAL GUIDANCE: Scenario: You've collected yield data from 5 management zones and need to determine if yields differ significantly between zones. You're unsure whether to use ANOVA, Kruskal-Wallis, or mixed effects model. Ellmer question: "I have continuous yield data (tonnes/ha) from 5 management zones with 30-50 observations per zone. Residuals appear non-normal based on Q-Q plot. Which statistical test is most appropriate for comparing means, and why?" Ellmer provides: (1) Explanation of assumptions for each test, (2) recommendation based on your data description (likely suggests Kruskal-Wallis for non-normal data), (3) R code for running chosen test and interpreting output, (4) guidance on post-hoc comparisons if overall test significant. This is MUCH faster than searching through statistics textbooks or multiple Stack Overflow threads. USE CASE 2 — CODE EXPLANATION: Scenario: You found complex dplyr pipeline online that does what you need, but you don't understand the syntax: yield_summary <- yield_data %>% filter(!is.na(yield)) %>% group_by(zone, year) %>% mutate(yield_scaled = scale(yield)) %>% summarize(across(starts_with("yield"), list(mean = mean, sd = sd))) Paste this into Ellmer with question: "Explain this code line by line, especially the 'across' and 'scale' parts." Ellmer breaks down: (1) filter removes NA yields, (2) group_by creates subgroups for summarization, (3) mutate with scale() creates z-scores within each group, (4) summarize with across() applies mean and sd to all columns starting with "yield", (5) resulting output structure. This builds your R skills by explaining patterns you can reuse, not just solving immediate problem. USE CASE 3 — DEBUGGING HELP: Scenario: Your script throws cryptic error: "Error in `summarize()`: Problem while computing `mean_yield = mean(yield)`. ℹ In argument: `mean_yield = mean(yield)`. Caused by error: ℹ argument is not numeric or logical: returning NA" Paste entire error message and relevant code into Ellmer: "I'm getting this error when trying to calculate mean yield. Here's my code: [paste code]. What's wrong?" Ellmer diagnoses: Yield column likely stored as character not numeric (common with imported CSV files that have text like "3.2 t/ha" instead of just numbers). Suggests: `yield <- as.numeric(yield)` to convert, and explains why error message was misleading. This is faster than trial-and-error debugging and teaches you to recognize this error pattern in future. USE CASE 4 — DATA EXPLORATION IDEAS: Scenario: You have spatial dataset (x, y coordinates + yield + soil EC) but unsure how to visualize relationships. Ellmer question: "I have point data with coordinates, yield, and soil EC. What are 3-5 effective visualizations to explore relationships and spatial patterns?" Ellmer suggests: (1) Scatter plot of yield vs soil EC colored by spatial clusters, (2) Spatial heatmap of yield using geom_tile, (3) Variogram to assess spatial autocorrelation structure, (4) Faceted plots comparing yield distribution across EC quartiles, (5) Interactive map using leaflet for detailed exploration. Provides code examples for each, allowing you to quickly generate suite of exploratory plots. USE CASE 5 — BEST PRACTICES: Question: "I have yield monitor data with some missing GPS coordinates (equipment briefly lost signal). Should I delete those rows, interpolate positions, or something else?" Ellmer provides nuanced answer considering: (1) Percentage of missing data (1% vs 20% requires different approaches), (2) Spatial continuity (if brief GPS dropout, interpolating reasonable; if random dropouts, not reliable), (3) Analysis goals (if calculating field average, missing coordinates less critical; if creating spatial map, critical), (4) Code examples for each approach with pros/cons. This domain-specific guidance is valuable because precision ag best practices aren't always documented in standard R resources.

Best Practices for LLMs in Data Analysis

  • Always review: Verify AI-generated code before execution
  • Understand output: Don't use code you can't explain
  • Protect credentials: Store API keys in .Renviron, never in scripts
  • Complementary tools: Copilot for speed, Ellmer for depth
  • Keep learning: LLMs accelerate, not replace, skill development
  • Update regularly: Keep packages current for best performance
As LLMs become standard tools in data analysis workflows, developing responsible usage habits is critical. PRINCIPLE 1 — ALWAYS REVIEW CODE: LLMs occasionally generate syntactically correct but logically wrong code. Example: Copilot suggests `filter(yield > 0)` to remove outliers, but this only removes negative values, not implausibly high values (typo errors like 85.3 t/ha instead of 8.53). ALWAYS run on small subset first, inspect output, verify logic matches intent. Risk scenarios: (1) Variable name close to but not exactly matching your data (e.g., suggests `yield_kg` but your column is `yield_tonnes`), (2) Function arguments with defaults different from what you need (e.g., `na.rm = TRUE` vs `na.rm = FALSE` changes result drastically), (3) Statistical tests with assumptions your data violates. PRINCIPLE 2 — UNDERSTAND OUTPUT: If LLM suggests unfamiliar function or approach, spend 5 minutes reading documentation before using. This builds mental model preventing future errors. Anti-pattern: Blindly copy-paste Ellmer suggestions into production analysis without understanding mechanism. Test yourself: After accepting Copilot suggestion, can you explain WHAT the code does and WHY it's correct approach? If not, investigate before proceeding. PRINCIPLE 3 — PROTECT CREDENTIALS: Real incident from 2023: Student committed API key to public GitHub repository. Key was scraped by bot within hours, used to make $200 worth of API calls in 24 hours before student noticed. GitHub notified Google who revoked key, but damage done. Prevention checklist: - ✅ API keys in .Renviron only - ✅ .Renviron in .gitignore (default behavior) - ✅ Before `git push`, run `git diff` to verify no secrets - ✅ Use GitHub secret scanning (automatically enabled for public repos) - ❌ Never paste API key in Slack, Discord, or email - ❌ Never hardcode as string in script - ❌ Never share .Renviron file itself PRINCIPLE 4 — COMPLEMENTARY TOOLS: Optimal workflow combines both: - Use Copilot continuously while coding (low friction, instant feedback) - Open Ellmer when stuck or need strategic guidance (higher latency but deeper help) - Use traditional documentation for foundational learning (LLMs supplement, not replace, reading official docs) Don't become dependent on any single tool — maintain ability to code without AI assistance for when you lack connectivity or tools unavailable. PRINCIPLE 5 — KEEP LEARNING: LLMs risk creating "competence ceiling" where you can implement solutions but not design them. Counter this by: (1) Deliberately practicing problems WITHOUT LLM assistance, (2) Studying LLM-generated code to understand patterns, (3) Taking on challenges slightly beyond your current skill level where LLMs scaffold learning. Goal: Use LLMs to accelerate journey from novice to expert, not replace expertise itself. PRINCIPLE 6 — UPDATE REGULARLY: LLM integrations evolve rapidly. Run `update.packages()` monthly to get improvements. Example: Ellmer version 0.1.0 (Jan 2025) had basic Gemini support; version 0.2.0 (Mar 2025) added streaming responses, chat export, and better error handling. Similarly, GitHub Copilot receives model updates automatically but RStudio extension requires updating through Tools → Check for Updates.

Open floor for questions. Common anticipated questions and responses: Q: "Will Copilot write my entire assignment for me?" A: No. Copilot assists with syntax and common patterns, but complex analysis requiring domain knowledge and critical thinking remains your responsibility. It accelerates implementation of ideas YOU conceive, not ideation itself. Q: "What if Copilot suggests wrong code?" A: This happens occasionally. Why reviewing and testing all suggestions is mandatory practice. Over time, you'll develop intuition for which suggestions to trust and which to verify carefully. Incorrect suggestions are often learning opportunities — figuring out WHY code was wrong builds debugging skills. Q: "Can I use these tools in exams?" A: Depends on exam format. Open-book exams with computer access: Yes. Closed-book written exams: No. Practical computing exams: Typically yes, but verify with exam instructions. Philosophy: We're teaching you to be effective data analysts in real world where these tools are standard, so artificially restricting them in assessment would be counterproductive. Q: "Does using LLMs count as cheating or plagiarism?" A: No, when used as analytical tools with appropriate citation. Yes, if you copy entire analysis plan from LLM and present as your own thinking without acknowledgment. Guideline: If LLM made intellectual contribution to your approach (not just syntax help), acknowledge it in methods section. Similar to citing statistics textbook that informed your choice of test. Q: "What if I can't get API key working?" A: Common issues: (1) Forgot to restart RStudio after editing .Renviron (solution: restart), (2) Typo in environment variable name (solution: check spelling is exactly GEMINI_API_KEY), (3) API key copied incorrectly (solution: regenerate new key). If still stuck, come to office hours or post in discussion forum — usually quick fix once we see error message. Q: "Are there alternatives to Gemini?" A: Yes. Ellmer package supports multiple LLM providers. Alternatives include Anthropic Claude (via API), OpenAI GPT (via API, costs money beyond free tier), or running local models like Llama2 (requires powerful computer). Gemini chosen for this course because: (1) Generous free tier, (2) Good performance on code and statistical questions, (3) Official support from Posit (RStudio developers). Q: "Should I still learn base R and tidyverse if LLMs can write code for me?" A: Absolutely yes. LLMs are tools that amplify your skills, not replacements for skills. Analogy: Calculator doesn't eliminate need to understand mathematics. You must understand programming fundamentals to: (1) Evaluate whether LLM suggestions are correct, (2) Debug when suggestions don't work, (3) Design overall analysis strategy that LLM then helps implement. Use LLMs to LEARN faster, not AVOID learning.
